💡 闭环设计原则（v5.9.2）
1. 教师归总部：教师统一由总部招募管理，线上直播授课覆盖全国所有门店班级，不外派、不挂代理商 TenantBind。
2. 门店由总部代开 + 4 步审批（v5.2 / v5.10）：代理商不自主开店，由运营经理在「+ 新增门店」录入草稿 → 代理老板确认（强校验地址）→ 运营经理审批 → 运营总监终审 → 自动激活门店 + 创建店长账号；任一环节驳回带说明回上一步；避免组织失控且确保地址准确（实际门店位置只有代理老板最清楚）。
3. 代理体系不参与教务（v5.2）：⛔ 老板不开店/不下单 · ⛔ 店长店员不下单/不代付/不排课；下单一律在家长端，排课一律在总部教学部。
4. 订单归门店 + 推广人（v5.2）：下单锁定门店（消课/分佣/结算的归属链），同时记录推广人（老板/店长/店员的分享码），推广业绩单独看板。
5. 合约审核 = 首款到账 一步合并（v5.6.2）：财务审核通过即视为已到账，同步建 Agent + 落 DeviceShipment，幂等 key=contractId。
6. 商品三层独立（v5.7.2）：CoursePackage / PhysicalItem 作为独立资源池，Goods 只引用不内嵌；家长端只认 Goods，底层构件不直售；底层禁用级联商品下架。
7. C 端订单按 PhysicalItem 拆单（v5.7.2）：一订单 N 条独立 DeviceShipment（sourceType=C_ORDER），可混合本地直发 + 代发；订单发货状态由聚合派生；体验账号仅 B 端生成。
8. 消课 = BIG 段 YES（v5.4）：1 节课生成 BIG + SMALL 两段 Attendance，只有大班段考勤 YES 才扣 1 课时，小班段仅考勤；避免"课时卖出但不上课"挂单。
9. 结算总部定期打款（v5.2）：月度 15 日前 / 季度 20 日前 / 年度年初一次性对公打款；代理端无提现入口。
10. 权限部门驱动（v5.6）：平台 6 部门（SALES/FINANCE/PURCHASE/WAREHOUSE/OPERATION/TEACHING）通过 functionType + PermissionMatrix 驱动菜单；"经理"由 Department.managerUserId 判定，没有独立角色；代理自动回收钩子对齐停用事件。

💡 v5.9.2 三合一模型核心变更：
① 三合一资源池：底层 3 个独立资源池 ——「课包管理」CoursePackage + 「实物管理」PhysicalItem + 「权益包管理」EntitlementPackage（v5.9.2 新），各自独立 CRUD 可被 N 个商品复用；顶层「商品管理」Goods 是家长端**唯一可下单的 SKU**。
② 自由组合：一个商品 = M 个课包 + N 个实物 + K 个权益包（M / N / K 都可 0..N，至少合计 ≥1 才能上架）；同一个底层资源可被多个商品引用复用。
③ 权益包语义：权益包 = App 使用时长（按天/月/年），购买后写 mall_student_entitlement(startAt + expireAt)，App 启动校验有效；同类型续费 expireAt 自动叠加。
④ 售卖前提：课包 / 实物 / 权益包**本身不直接对家长售卖**，必须被包装成商品（SKU）后才能卖。
⑤ 引用不内嵌：商品里存 coursePackageIds[] + physicalItemIds[] + entitlementPackageIds[]，不拷贝快照；底层改价时商品展示自动联动（已下订单锁定订单时快照不变）。
⑥ 展示增强：商品层才有多图（coverImages[]）+ 富文本详情（detailHtml）+ 单选分类 + 多标签；底层资源只维护自身业务字段。
⑦ 状态传导：任一底层资源被下架 / 删除 → 引用它的商品自动 OFFLINE 并站内信通知运营。
⑧ 草稿状态：商品 goodsStatus=DRAFT 运营自己可见；3 个底层资源也各自有 status=DRAFT/ENABLED/DISABLED。
⑨ 不做库存：商品层不存库存；超卖保护留到 §3.4 仓储发货环节由仓储拒单。
